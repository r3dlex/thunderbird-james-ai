# Features

## Contextual Email Chat

**Entry point**: `message_display_action` popup (Tab 1 in the UI)

When the user opens the chat panel while viewing an email:

1. The UI sends a `getDisplayedMessage` request to the background script
2. Background reads the displayed message via `messenger.messageDisplay.getDisplayedMessage()`
3. Background builds thread context using `get_thread` logic (references chain)
4. The system prompt is assembled from a template:

```
You are Corvus, an AI email assistant. You have access to the following context:

--- Current Email ---
From: {from}
To: {to}
Date: {date}
Subject: {subject}

{body}

--- Thread ({threadCount} messages) ---
{threadSummary}

You can use the following tools to search, move, tag, draft, and read emails.
Respond concisely. When drafting replies, match the tone of the conversation.
```

5. User messages are sent to the AI router with the full context and tool definitions
6. Responses stream back via `runtime.Port` with `content_delta` messages
7. Conversation history is held in memory for the popup session lifetime (cleared on popup close)

**Capabilities in this view**:
- Summarize the current email or thread
- Draft replies (creates draft, never sends)
- Search related emails
- Move or tag the current message
- Answer questions about the email content
- AI-driven classification: the LLM can classify the email by type, priority, or topic based on its content analysis
- AI-driven auto-reply: when configured via rules, the extension can automatically draft and optionally send replies to emails matching certain classifications (see Rules Engine below)

## General Assistant

**Entry point**: `action` popup (Tab 2 in the UI)

Not tied to any specific email. Supports multi-step operations across the mailbox:

- Search and filter emails by complex criteria
- Batch move, tag, or organize messages
- Create new drafts from scratch
- Summarize unread messages in a folder
- AI-driven email filtering and search

**Batch confirmation**: Operations affecting more than 10 messages require explicit user confirmation. The UI displays a confirmation card listing affected messages before executing.

**Multi-step flow**: The AI router's tool loop allows the assistant to chain operations (e.g., search for emails matching criteria, then move the results to a folder). Each step's tool results feed into the next iteration.

## Compose Assistant

**Entry point**: `compose_action` popup

Activated from the compose window. Reads current draft state via `messenger.compose.getComposeDetails()`.

**Quick-action cards** (displayed as clickable cards in the UI):

| Action | Instruction |
|--------|-------------|
| Concise | Shorten the draft while preserving key points |
| Formal | Rewrite in formal/professional tone |
| Grammar | Fix grammar and spelling errors |
| Translate EN/DE | Translate between English and German |

**Custom instructions**: Free-text input for arbitrary rewrite requests.

**Behavior**: The assistant reads the current compose body, applies the requested transformation via the AI provider, and writes the result back via `messenger.compose.setComposeDetails()`. It modifies the draft only -- it never calls `messenger.compose.sendMessage()`.

## Automated Rules Engine

Rules automate actions on incoming mail using a condition/action model with optional AI-powered classification.

### Rule Structure

```typescript
interface CorvusRule {
  id: string
  name: string
  enabled: boolean
  conditions: CorvusCondition[]
  conditionLogic: "all" | "any"
  actions: CorvusAction[]
  createdAt: string
  updatedAt: string
}

interface CorvusCondition {
  field: "from" | "to" | "subject" | "body" | "folder" | "tags" | "ai_classification"
  operator: "contains" | "equals" | "startsWith" | "matches" | "ai_matches"
  value: string
}

interface CorvusAction {
  type: "move" | "tag" | "notify" | "auto_reply" | "mark_read" | "flag"
  params: Record<string, string>
}
```

### Evaluation

- Rules are evaluated on `messenger.messages.onNewMailReceived` events
- Additionally, a scheduled alarm (`messenger.alarms`) triggers evaluation every 5 minutes to catch messages that arrived while the extension was inactive
- Conditions are matched in order; `conditionLogic` determines AND/ANY semantics
- When `field` is `ai_classification`, the engine sends the email content to the AI provider with a classification prompt and checks the result against the condition's `value`
- Rule evaluation target: under 50ms per message for non-AI conditions

### Auto-Reply

The `auto_reply` action creates and sends a reply via the AI provider:

- The reply content is generated by the AI using the rule's `params.prompt` as instructions
- Deduplication: tracks replied-to `headerMessageId` values in storage to prevent duplicate auto-replies
- Only rules explicitly created by the user can trigger auto-reply -- no implicit or default auto-reply behavior
- Auto-replies use `messenger.compose.beginReply()` followed by `messenger.compose.sendMessage()`

### AI-Assisted Rule Creation

Users can describe rules in natural language (e.g., "Move newsletters to the Newsletters folder"). The AI parses the description into a `CorvusRule` structure. The generated rule is shown to the user for review before activation.

### LLM-Powered Classification Rules

Rules can use `ai_classification` conditions with the `ai_matches` operator to classify incoming emails:

- The engine sends the email to the AI provider with a classification prompt derived from the condition's `value`
- The AI returns a classification label
- If the label matches, the rule's actions execute
- This enables rules like "If the AI classifies this as a support request, auto-reply with acknowledgment and tag as support"
- Classification results are cached per `headerMessageId` to avoid redundant AI calls
